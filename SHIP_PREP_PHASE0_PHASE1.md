# /ship 前任务清单（Phase 0 / Phase 1）

目标：在进入 `/ship` 之前，把 Phase 0（验证）和 Phase 1（单竖切片实现）做到可交付、可验收、可回滚。

范围基线（已确认）
- 单竖切片：`1 个稳定数据源 + 快照标准化 + 审计链 + 周报 API`
- 不做：多数据源并行上线、移动端、扩展分析功能
- 关键约束：不允许静默失败，必须有 `stale` 可见告警

---

## Phase 0（T+0 ~ T+6）：需求与门禁闭合

### Day 1：锁定候选数据源与访谈名单
- [ ] 确认 2 个候选数据源（必须满足：稳定 API、可拿到沙盒、鉴权方式明确）
- [ ] 建立 5 位目标用户访谈名单（最低门槛 3 位）
- [ ] 输出《数据源可行性一页纸》

验收标准
- 候选源有明确 API 文档链接与鉴权方式
- 访谈名单包含角色、联系方式、可访谈时间

### Day 2：完成首批访谈（2 人）
- [ ] 访谈用户 #1、#2
- [ ] 记录 3 个量化点：每周核对耗时、最近一次决策延迟、愿付费区间
- [ ] 记录现有流程中的失败点（超时、对账差异、重复核对）

验收标准
- 每个受访者都有结构化访谈记录
- 至少提炼 3 条可量化痛点

### Day 3：完成访谈（再 2 人）+ 周报字段初稿
- [ ] 访谈用户 #3、#4
- [ ] 产出周报字段初稿（<=8 项）
- [ ] 给每个字段写清楚定义与来源

验收标准
- 字段清单含“定义、计算方式、来源、异常处理”

### Day 4：补齐第 5 人访谈 + 决策合并
- [ ] 访谈用户 #5（若未凑齐，至少达到 3 位并标注风险）
- [ ] 合并访谈结果，输出优先级排序
- [ ] 锁定 G1/G2：首批数据源与周报字段

验收标准
- 已形成可执行优先级（不是意见列表）
- G1/G2 明确“通过/不通过”

### Day 5：技术门禁闭合（G3/G4）
- [ ] 定义并发同步策略：并行拉取 + 版本化聚合快照
- [ ] 定义角色模型：`admin` / `operator` / `viewer`
- [ ] 定义 stale 告警策略（用户端可见）

验收标准
- 有明确流程图（请求、后台任务、快照、告警）
- 权限矩阵可直接转为 API 鉴权规则

### Day 6：Phase 0 出口评审（Go/No-Go）
- [ ] 开 Phase 0 出口会（30-45 分钟）
- [ ] 结论：是否进入 Phase 1（Go/No-Go）
- [ ] 若 No-Go，记录阻塞项与修复计划

Go 条件（全部满足）
- G1~G4 全闭合
- 访谈 >=3（目标 5）且指标可量化
- 周报字段 <=8 且定义完整
